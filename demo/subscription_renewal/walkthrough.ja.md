# Walkthrough: Feature ReviewのPRを`$socratic`でレビューする

このファイルは、実際のセッションで「何が起きるか」の実況です。どういう状態で実行すると、ターミナルに何が出て、何を聞かれ、どう答えると、何が返ってくるかを順に示します。

## 1. 状態

PRがサブスクリプション更新機能`renew()`を追加しました。テストは2件だけ——「明確に期限切れなら拒否」と「有効なら成功を返す」——で、どちらもGreenです。期限境界・課金・リトライの仕様はコードからは確定できません。

## 2. 実行

```text
> $socratic このPRをレビューして
```

## 3. ターミナル(調査フェーズ)

```text
Socratic: Feature Reviewとして進めます。

  調査中...
  - 変更: subscription/renew.py (+31), tests/test_renew.py (+18)
  - 既存テスト2件を実行開始時点のSnapshotとして記録
  - 依存の分類: charge(決済Gateway) = プロセス外・管理外
    -> 送信内容と回数をアプリケーション境界で検証します
  - リポジトリからは確定できない仕様が2件あります
```

## 4. 質問(構造化UI)

回答者は仕様オーナーです。各選択肢に、選んだ場合のテストへの影響が1文で付きます。

```text
┌ 質問 1/2 ── 期限境界 ─────────────────────────────────────┐
│ 契約終了日当日の更新を許可しますか?                        │
│                                                            │
│   1. 許可する                                              │
│      終了日当日を有効期間に含め、当日の更新成功を固定します │
│ ▸ 2. 拒否する                                              │
│      終了日から期限切れとし、当日の更新拒否を固定します    │
│   3. その他(自由入力)                                      │
└────────────────────────────────────────────────────────────┘
あなた: 2. 拒否する

┌ 質問 2/2 ── 課金 ─────────────────────────────────────────┐
│ 成功した更新は、リトライを通じて何回課金すべきですか?      │
│                                                            │
│ ▸ 1. 正確に1回 (推奨)                                      │
│      リトライでの再課金をテストで禁止します                │
│   2. リトライごとに課金してよい                            │
│      冪等性のテストを追加しません                          │
│   3. その他(自由入力)                                      │
└────────────────────────────────────────────────────────────┘
あなた: 1. 正確に1回
```

## 5. ターミナル(検証フェーズ)

```text
Socratic: 回答をIntent Contractへ記録しました (DEC-001, DEC-002)。

  Disposable環境でテストを設計・証明中...
  - 提案テスト3件を作成
  - 3つの重大インシデントをMutationとして再現
  - 元コードで成功 / 各Mutantで失敗 を確認 (双方向証明)
```

## 6. 結果(正準の4ブロック)

```text
Socratic Review

Review This:
  (人間の判断は残っていません)

We Verified:
  ✓ 実行開始時点で既存のテスト2件を評価
  ✓ 期限切れ後の更新は、Disposable環境で提案・証明済みのテストが拒否を検知
  ✓ 未課金での成功は、同じく提案・証明済みのテストが検知
  ✓ リトライの二重課金は、同じく提案・証明済みのテストが検知
  ✓ 今回のReview-only実行中、Working Treeは不変

Still at Risk:
  △ 提案テスト3件は未適用
    上の保護は適用まで永続化されません

Copy-ready Comments:
  (なし — 仕様は本セッションで確定済み)
```

## 7. 後処理の質問

```text
┌ 証明済みテストの扱い ──────────────────────────────────────┐
│ ▸ 1. テストを適用する                                      │
│      検証済みPatchをWorking Treeへ適用し、証明を再実行します│
│   2. Patchを出力する                                       │
│      Working Treeは変更せず、Patch全文を表示します         │
│   3. 破棄する (無回答時の既定)                             │
└────────────────────────────────────────────────────────────┘
あなた: 1. テストを適用する
```

## 8. 最終状態

```text
Socratic: 引き渡しPatchのHashと前提条件を検証し、テスト3件を適用しました。
  元コードでの成功と各Mutantでの失敗を再証明済みです。

Socratic Review (更新)

We Verified:
  ✓ 3つの重大インシデントを、明示依頼後に今回の実行が適用したテストが検知
  ✓ 変更されたWorking Tree: tests/test_renew.py

Still at Risk:
  (残存なし)

実行時の成果物(Intent Contract / Report)はどうしますか?
  保存しない(既定) / ローカルに保存 / Markdownとして出力
あなた: 保存しない
```

コミットやPushは行われません。適用されたテストをどう扱うかは、あなたの通常のGitフローに残ります。

## この題材を自分で動かす

このディレクトリの実行可能デモは、上のセッションの「エンジン部分」——弱いテストが3つのMutantを見逃し、確定仕様のテストが全部検知すること——を10秒で再現します。

```bash
python3 -m demo.subscription_renewal.run_demo
```
