THE PROBLEM

テストはGreenでも安心できない

テストがGreenでも、それだけでは変更意図や期待する振る舞いは証明できません。 AIが大量にコードを書くほどDiffは大きくなり、全行を追っても「なぜ変えたか」はコードだけでは確定できません。

Socraticは変更前後を観測可能な振る舞いで比較し、確定できない変更意図だけを人間へ質問。 回答から期待値とテストのOracleを明確にします。

HOW IT WORKS

Ask. Clarify. Verify.

変更を比較し、意図を明確にし、テストで確かめる。 3つのSkillが、曖昧な不安を判断可能な材料へ変えます。

01ORCHESTRATE

Socratic

比較して、論点を絞る

BaseとHeadの変更前後を比較し、仕様判断・振る舞い差・テスト不足を抽出。レビュー全体を組み立てます。

BEFOREBase ↔ AFTERHead
COMPARE
仕様判断 振る舞い差 テスト不足
ORCHESTRATE
REVIEW PLAN人間が決めるべき論点
OUTCOME 大きなDiffが、少数の判断事項になる
02ELICIT

Maieutic

意図を引き出す

実装だけでは確定できない期待を、仕様オーナーが答えられる具体的な質問へ変換します。

未確定の仕様→人間への質問→Intent Contract
OUTCOME 曖昧な不安が、答えられる仕様質問になる
03REFUTE

Elenchus

反証して確かめる

BaseとHeadへ同じテストを実行し、さらにバグを注入してテストの検知能力を証明します。

Base / Head→Mutation→検知の証拠
OUTCOME 「Green」が、バグを検知できる証拠になる

USE CASES

Choose your review mode.

01 / SPECIFICATION

Feature Review

新しい振る舞いの「正しさ」を決める

境界条件や例外時の期待を質問にし、仕様オーナーの回答をIntent Contractとテストへ結びます。

● ● ●socratic · intent check
› Use $socratic to review this change

  • いいえ同日は更新できません
  • 別の仕様を入力する…
↑↓ 移動enter 決定
02 / REGRESSION

Refactor Guard

リファクタリングの振る舞い差を見つける

BaseとHeadへ同じBehavior Testを実行。差があれば、意図した変更か回帰かを人間へ確認します。

BASE→Behavior Test→観測結果 A
HEAD→Behavior Test→観測結果 B
COMPARE OUTPUTSBehavior Diff?
03 / TEST QUALITY

Test Assessment

AIが追加したテストの「検知力」を測る

同じリスクMutationを既存・変更後のテストへ当て、増えた保護と、まだ守られていないリスクを分離します。

BASE TESTS＋ MUTANT→FAIL ✓
HEAD TESTS＋ MUTANT→FAIL ✓
MUTANT KILLEDRisk detected

SAFE BY DEFAULT

安全に調べる。勝手に変えない。

DEFAULT: REVIEW-ONLYPR / GitHub / Working Treeへの書き込みなし

01

Review-onlyが既定

PR、GitHub、RepositoryのWorking Treeへ書き込みません。

02

隔離環境で検証する

比較テストとMutationはDisposableな環境で実行し、Headの本番コードを変更しません。

03

Git操作を自動化しない

Stage、Commit、Push、Branch切替、PR作成、コメント投稿を行いません。

04

書き込みは明示選択時だけ

証明済みテストも、ユーザーが「Apply tests」を選ぶまではWorking Treeへ追加しません。

WHY SOCRATIC?

問い、引き出し、反証する。

名前はソクラテスの問答法に由来します。分かっていないことを自覚して問いを立て、 対話によって相手の中から考えを引き出し、反駁によって主張を吟味する。 その進め方を、コードレビューの手順にしています。

01 / INQUIRE

Socratic

問いを立てる

コードだけでは分からないことを認め、人間が判断すべき変更意図を見つけます。

02 / ELICIT

Maieutic

考えを引き出す

産婆術のように対話を通じて期待する振る舞いを言葉にし、Intent Contractへ記録します。

03 / REFUTE

Elenchus

反証して確かめる

意図と異なる振る舞いをあえて作り、テストが実際に検知できるかを確かめます。

RESEARCH ROOTS

Research behind
the workflow.

Socraticは、変更時にバグを捕まえるテストと、自然言語の意図から振る舞いの変種を作る Mutation Testingという、相互補完的な研究を接続しています。

2025 / JIT TESTING

Harden and Catch for Just-in-Time Assured LLM-Based Software Testing

将来の回帰を防ぐHardening Testと、変更で生じた欠陥を着地前に捕まえるCatching Testを定義し、Just-in-Timeなテスト生成の研究課題を整理した基礎論文です。

2026 / INDUSTRIAL SCALE

Just-in-Time Catching Test Generation at Meta

Metaの大規模バックエンドで22,126件の生成テストを分析。変更を理解する生成と判定器により、人間の確認負荷を抑えながら本番前に重大な欠陥を捕捉できることを報告しています。

Read the paper ↗
2026 / INTENT MUTATION

Intent-Based Mutation Testing: From Naturally Written Programming Intents to Mutants

自然言語で書かれた意図を少し変え、その意図を実装するMutantを生成する手法です。29プログラムの評価で、構文ベースMutationとは異なる欠陥クラスを補完できる可能性を示しています。
